Q：为什么 HashMap 不是线程安全的？在多线程环境下使用会有什么问题？
A：HashMap 的线程不安全主要体现在两个地方：  
1. **数据覆盖**：当多个线程同时执行 put 操作时，可能会因为计算出的哈希槽位相同而导致前一个线程写入的值被后一个覆盖。  
2. **死循环**（JDK 1.7 及之前）：扩容时采用头插法，多线程并发扩容可能形成环形链表，导致 get 操作进入死循环。  
3. **数据丢失**：多线程同时 put 时，size 计数不是原子操作，可能导致 size 与实际元素数量不一致。  
因此，在多线程环境下应使用 ConcurrentHashMap 或 Collections.synchronizedMap 来保证线程安全。

Q：volatile 是如何保证可见性的？底层原理是什么？
A：volatile 保证可见性的核心是 **内存屏障**。  
1. 当写一个 volatile 变量时，JVM 会在写操作后插入一个 **StoreStore 屏障**，并强制将当前线程的工作内存中的值刷新到主内存。  
2. 当读一个 volatile 变量时，JVM 会在读操作前插入一个 **LoadLoad 屏障**，强制使其他线程的缓存行失效，从而从主内存中重新读取。  
3. 底层 CPU 实现通常使用 **MESI 缓存一致性协议**，配合 lock 前缀指令（x86）来保证缓存同步。

Q：synchronized 和 ReentrantLock 有什么区别？如何选择？
A：区别如下：
1. **锁实现**：synchronized 是 JVM 层面的锁，ReentrantLock 是 API 层面的锁（通过 AQS 实现）。  
2. **功能**：synchronized 仅支持非公平锁；ReentrantLock 支持公平锁、可响应中断、可超时获取锁，还支持多个条件变量（Condition）。  
3. **性能**：在 JDK 1.6 之后，synchronized 经过锁升级优化（偏向锁→轻量级锁→重量级锁），性能与 ReentrantLock 差距很小。  
选择建议：如果只需要基本同步，优先使用 synchronized；如果需要高级特性（如公平锁、可中断），则使用 ReentrantLock。

Q：一个侦探在案发现场发现一张纸条，上面写着“B=2, A=1, R=18, 死亡时间=?”。已知凶手是用字母顺序来隐藏信息的，请问死亡时间是多少？
A：1. 将字母转换为数字：B=2, A=1, R=18。  
2. 观察规律：可能是将字母按某种运算得到时间。  
3. 尝试求和：2+1+18=21，但21不是常见时间表示。  
4. 尝试乘积：2×1×18=36，也不像时间。  
5. 另一种思路：凶手可能用字母在单词中的位置暗示时间。单词“BAR”在字母表中的顺序分别是2、1、18，如果当作时间“2:18”或“18:2”都不合理。  
6. 考虑将数字作为时和分：2:18 是凌晨2点18分，但纸条写的是“死亡时间=?”，也许凶手用简单的规律：B=2，A=1，R=18，合起来是“2点18分”。  
7. 最终推理：凶手将三个字母对应的数字直接拼接成时间“2:18”，因此死亡时间为凌晨2点18分。

Q：一个逻辑题：甲、乙、丙三人，一个只说真话，一个只说假话，一个有时说真话有时说假话。你问甲：“乙是那个有时说真话的人吗？”甲回答：“是。”问乙：“丙是那个只说真话的人吗？”乙回答：“是。”问丙：“甲是那个只说假话的人吗？”丙回答：“不是。”请问谁是什么类型？
A：1. 用符号表示：T（只说真话），F（只说假话），R（随机）。  
2. 假设甲是T，则甲说“乙是R”为真，故乙=R。那么乙回答“丙是T”为真（因为乙是R，但R不一定说真话，这里矛盾：如果乙是R，他的回答真假不定，但我们已经假设甲为T，乙为R，乙说“丙是T”可以是假的，也可以是真的，无法唯一确定）。  
3. 假设甲是F，则甲说“乙是R”为假，所以乙不是R，那么乙是T或F。乙回答“丙是T”如果乙是T，则丙=T，但甲已经是F，丙不能也是T，矛盾；如果乙是F，则“丙是T”是假，所以丙不是T，丙可能是F或R。丙回答“甲是F”，因为甲是F，丙说“不是”即否定“甲是F”，所以丙说的是假话，因此丙是F（因为R不能说假话？注意R可以随意说，但此处丙回答“不是”，实际甲是F，所以丙说的是假话，如果丙是F，则说假话符合；如果丙是R，他可能说假话，但概率问题）。此时假设甲=F，乙=F，丙=F？但三人不能都是F。所以这条路径排除。  
4. 假设甲是R，那么甲的回答无参考价值。此时乙和丙在T和F中。乙说“丙是T”，如果乙=T，则丙=T，那么三人中甲=R，乙=T，丙=T，但有两个T，不满足条件；如果乙=F，则“丙是T”是假，所以丙不是T，则丙=F，于是甲=R，乙=F，丙=F，又是两个F，也不满足。  
5. 重新审视：可能我之前的推理假设有问题。实际标准答案是：甲=R，乙=T，丙=F。验证：甲说“乙是R”是假（因为乙=T），作为R可以说假话，没问题。乙说“丙是T”是假（丙=F），但乙=T应该说真话，这里矛盾。所以答案可能是甲=F，乙=R，丙=T？验证：甲=F说“乙是R”假，所以乙不是R，则乙=T或F；乙是R（假设），乙说“丙是T”作为R可以是假，丙不是T；丙=T说“甲是F”真，甲=F正确。符合。  
因此答案：甲是只说假话，乙是随机，丙是只说真话。

Q：你被困在一个房间里，有两个门，一个通向自由，一个通向死亡。两个守卫，一个只说真话，一个只说假话。你只能问其中一个守卫一个问题，然后选择一扇门。你应该问什么？
A：经典逻辑谜题：你只需要问其中任何一个守卫：“如果我问另一个守卫哪扇门是自由的，他会指哪扇？”然后你选择相反的门。  
推理：  
1. 假设守卫A是真话者，守卫B是假话者。你问A：“B会指哪扇门？”A知道B会指死亡之门（因为B假话），所以A会指向死亡之门。此时你选择相反的门，即自由门。  
2. 假设守卫A是假话者，守卫B是真话者。你问A：“B会指哪扇门？”A会撒谎，B实际会指自由门，A撒谎所以指死亡门。同样你选择相反的门，即自由门。  
因此无论问谁，你都会得到指向死亡门的答案，选择另一扇即可。

Q：一个有趣的伦理困境：一辆有轨电车失控，前方轨道上有5个被绑的人，你可以拉杆让电车转向另一条轨道，但那条轨道上有1个人。你会拉杆吗？为什么？
A：  
- **Ti（逻辑型）**：从功利主义出发，拉杆能救5个人，牺牲1个人，总伤害最小，所以应该拉。  
- **Fe（情感型）**：考虑到不直接动手杀人，拉杆相当于主动选择牺牲那1个人，可能带来心理负担，但为了多数人，仍然会拉。  
- **Fi（价值观型）**：认为每个人的生命都同等重要，不能为了多数牺牲少数，因此不拉。  
- **Ne（发散型）**：可能会提出其他方案，比如跳下去救人、或者找其他方法，不局限于两个选择。  
- **Te（效率型）**：应该拉杆，因为这样能最大程度减少损失，符合效率原则。  
- **Si（经验型）**：根据以往类似案例，多数人会选择拉杆，因为牺牲一个人拯救更多人被普遍接受。
- **Se（行动型）**：立即拉杆，因为每多犹豫一秒，就可能错过最佳时机。  
- **Ni（洞察型）**：这个困境反映了人类在紧急状态下对生命价值的权衡，拉杆与否都揭示了更深层的道德原则，长远看可能影响社会对紧急避险的立法。  
这个问题没有标准答案，不同风格反映不同的价值判断。

Q：一个游戏策略：在“石头剪刀布”游戏中，如果对手总是出石头，你应该出什么？
A：1. 观察对手模式：固定出石头。  
2. 推理：石头克制剪刀，被布克制。  
3. 因此你应该出布，这样可以赢。  
4. 进一步推理：如果对手意识到你的策略，可能会改变，所以需要动态调整。这展示了博弈论中的“最优反应”概念。

Q：生活决策：你收到两个工作offer，一个工资高但加班多，一个工资低但工作生活平衡，如何选择？
A：1. 列出影响因素：收入、健康、家庭时间、职业发展、个人兴趣。  
2. 加权打分：对每个因素分配权重（如收入0.3，健康0.2，家庭0.3，发展0.1，兴趣0.1），对两个offer分别打分。  
3. 计算总分，选择高者。  
4. 如果权重主观，可以用“最小最大后悔值”法或咨询亲友。  
最终决策取决于个人价值观（Fi/Fe）和逻辑分析（Ti/Te）的结合。

Q：什么是“锁升级”？synchronized 在 JDK 1.6 之后做了哪些优化？
A：1. 锁升级是指从无锁 → 偏向锁 → 轻量级锁 → 重量级锁的过渡过程。  
2. 偏向锁：当只有一个线程访问同步块时，JVM 会将锁对象头标记为偏向模式，避免 CAS 操作，提高单线程性能。  
3. 轻量级锁：当有另一个线程尝试获取锁时，偏向锁会升级为轻量级锁，通过自旋（自适应自旋）尝试获取锁，避免线程阻塞。  
4. 重量级锁：当自旋超过一定次数或并发激烈时，锁升级为重量级锁，线程会进入阻塞队列，由操作系统管理。  
5. 这种优化使得 synchronized 在单线程和轻度竞争场景下性能大幅提升，接近无锁。

Q：线程池的核心线程数、最大线程数、任务队列如何协同工作？
A：1. 当提交一个新任务时，如果当前运行的线程数小于 corePoolSize，即使有空闲线程，也会创建新线程执行任务。  
2. 如果当前线程数大于等于 corePoolSize，则尝试将任务放入阻塞队列（workQueue）。  
3. 如果队列已满，且当前线程数小于 maximumPoolSize，则继续创建新线程执行任务。  
4. 如果队列已满且线程数已达 maximumPoolSize，则执行拒绝策略。  
5. 当线程数超过 corePoolSize 时，空闲线程会在 keepAliveTime 后被回收，直到线程数降回 corePoolSize。

Q：什么是“回表”？如何通过覆盖索引避免回表？
A：1. 在 InnoDB 存储引擎中，普通索引的叶子节点存储的是主键值（而非完整行数据）。  
2. 当通过普通索引查询时，先找到主键值，再通过主键索引去查找整行数据，这个过程就是“回表”。  
3. 如果查询需要的所有字段都已经包含在索引中（即“覆盖索引”），则无需回表，直接返回索引中的值即可。  
4. 例如，在表 user(name, age) 上建立索引 (name, age)，查询 `select name, age from user where name='Tom'` 就不需要回表，因为 name 和 age 都在索引中。

Q：介绍一下 JVM 的“逃逸分析”及它能做什么优化？
A：1. 逃逸分析是 JIT 编译器的一种技术，分析对象的作用域是否逃逸出当前方法或线程。  
2. 如果对象没有逃逸（即只在方法内部使用），JVM 可以执行以下优化：  
   a) **栈上分配**：将对象分配在栈上，随方法结束自动销毁，减轻 GC 压力。  
   b) **标量替换**：将对象拆分为其成员变量（标量）并直接使用寄存器或栈变量，避免对象创建。  
   c) **同步消除**：如果对象不会逃逸，则对其加锁的操作可以被消除，提高并发性能。  
3. 逃逸分析默认在 Server 模式下开启，通过 `-XX:+DoEscapeAnalysis` 控制。

Q：什么是“最终一致性”？在分布式系统中如何实现？
A：1. 最终一致性是分布式系统的一种一致性模型，它允许数据在一段时间内不一致，但保证最终所有副本会达到一致状态。  
2. 实现手段包括：  
   a) **异步复制**：主节点写入后立即返回，从节点异步同步数据。  
   b) **Quorum 机制**：通过多数派读写来保证最终一致（如 Dynamo）。  
   c) **Gossip 协议**：节点间周期性地交换信息，逐步传播更新。  
   d) **基于时间戳或版本号**的冲突解决（如最后写入获胜）。  
3. 适用于对实时性要求不高但需要高可用和扩展性的场景，如社交动态、购物车等。

Q：什么是“虚假共享”（False Sharing）？如何避免？
A：1. 多核 CPU 中，每个核心有自己的缓存行（通常 64 字节）。  
2. 当两个线程分别修改同一缓存行中的不同变量时，即使变量本身独立，也会导致缓存行失效，迫使另一个核心重新加载，造成性能下降，这就是“虚假共享”。  
3. 避免方法：  
   a) 通过填充（padding）使变量独占缓存行，如 Java 中 @Contended 注解。  
   b) 使用线程局部变量（ThreadLocal）减少共享。  
   c) 调整数据结构，让频繁写操作的变量分散到不同缓存行。  
4. 示例：ConcurrentHashMap 的 CounterCell 就使用了缓存行填充来避免虚假共享。

Q：介绍一下“双亲委派模型”及其好处。
A：1. 双亲委派模型是 Java 类加载器的一种工作模式：当一个类加载器收到类加载请求时，先委派给父类加载器加载，只有父类无法加载时才自己尝试。  
2. 层次结构：启动类加载器（Bootstrap）→ 扩展类加载器（Extension）→ 应用程序类加载器（Application）。  
3. 好处：  
   a) 避免类的重复加载，保证核心类库（如 java.lang.Object）的唯一性。  
   b) 安全性，防止用户自定义类冒充核心类库（因为核心类优先被父加载器加载）。  
4. 破坏双亲委派模型的情况：如 Tomcat 等容器需要隔离不同应用的类，会自定义类加载器。

Q：什么是“一致哈希”（Consistent Hashing）？在分布式缓存中如何应用？
A：1. 一致哈希是一种分布式哈希算法，用于将数据均匀分布到节点上，并在节点增减时最小化数据迁移。  
2. 核心思想：将节点和键都映射到一个哈希环上，键顺时针找到的第一个节点即为存储位置。  
3. 节点增减时，只影响相邻节点间的数据，迁移量约为 1/N（N 为节点数）。  
4. 应用：Redis Cluster、Amazon Dynamo、Cassandra 等都使用了类似思想。  
5. 虚拟节点：为解决节点分布不均，引入虚拟节点，一个物理节点对应多个虚拟节点，使数据更均匀。

Q：Java 中的“内存屏障”是什么？volatile 是如何利用它实现可见性的？
A：1. 内存屏障（Memory Barrier）是一种 CPU 指令，用于强制刷新内存缓存，确保指令顺序。  
2. volatile 写操作前插入 StoreStore 屏障，写后插入 StoreLoad 屏障，强制将工作内存数据刷新到主内存。  
3. volatile 读操作前插入 LoadLoad 屏障，读后插入 LoadStore 屏障，使当前线程的缓存失效，从主内存重新读取。  
4. 这些屏障保证了 volatile 变量的读写不会重排序，且对其他线程立即可见。

Q：介绍下“AQS”的实现原理，它是如何管理等待队列的？
A：1. AQS（AbstractQueuedSynchronizer）是 Java 并发包中锁和同步器的骨架，内部维护一个双向链表（CLH 队列）来管理等待线程。  
2. 每个节点（Node）保存线程、等待状态（CANCELLED、SIGNAL 等）。  
3. 获取锁时，通过 CAS 尝试修改 state，失败则加入队列并阻塞（LockSupport.park）。  
4. 释放锁时，唤醒后继节点（LockSupport.unpark）。  
5. 公平锁：按照队列顺序获取；非公平锁：新线程可以抢占。  
6. ReentrantLock、Semaphore、CountDownLatch 等都基于 AQS 实现。

Q：什么是“补偿事务”？在微服务中如何实现最终一致性？
A：1. 补偿事务（TCC：Try-Confirm-Cancel）是一种分布式事务解决方案。  
2. 核心思想：将业务操作拆分为 Try（预留资源）、Confirm（确认执行）、Cancel（回滚）三个阶段。  
3. 先执行所有参与方的 Try 操作，如果全部成功，再执行 Confirm；否则执行 Cancel。  
4. 优点：不锁定资源，并发性高；缺点：开发复杂，需业务支持可回滚。  
5. 其他实现方式：SAGA（通过事件驱动，定义补偿操作）、本地消息表、最大努力通知等。

Q：经典谜题：一个房间里有 3 个开关，对应隔壁房间的 3 盏灯。你只能进入隔壁房间一次，如何确定哪个开关控制哪盏灯？
A：1. 打开第一个开关，等待几分钟后关闭；打开第二个开关，保持打开；第三个开关不动。  
2. 然后进入隔壁房间：  
   - 亮着的灯由第二个开关控制。  
   - 不亮但发热的灯由第一个开关控制（因为之前打开过）。  
   - 不亮且冷的灯由第三个开关控制。  
3. 推理链：利用灯的热量作为额外信息，打破一次观察的限制。

Q：一个逻辑题：三个逻辑学家走进酒吧，酒保问：“每个人都想喝啤酒吗？”第一个说：“我不知道。”第二个说：“我不知道。”第三个说：“是的，我们都要。”请问他们是如何推理的？
A：1. 如果第一个人自己不想喝，那么他会知道“并非每个人都想喝”，从而回答“不是”。但他回答“不知道”，说明他自己想喝（但他不确定别人）。  
2. 第二个人听到第一个人说“不知道”，推断出第一个人想喝。如果第二个人自己不想喝，那么他知道“并非每个人都想喝”，就会回答“不是”。但他也回答“不知道”，说明他自己也想喝。  
3. 第三个人听到前两个都说“不知道”，推断出前两人都想喝。如果他自己不想喝，他会回答“不是”；但他回答“是的，我们都要”，说明他自己也想喝。  
4. 因此，三人实际上都想喝。  
5. 此谜题体现了共同知识和递归推理。

Q：囚徒困境的变种：两个囚犯被分开审讯，如果两人都沉默，各判 1 年；一人招供一人沉默，招供者释放，沉默者判 5 年；两人都招供，各判 3 年。如果你是其中一个囚犯，会怎么选择？为什么？
A：1. 从个人理性角度，无论对方怎么做，招供都是占优策略（5 年 vs 1 年；0 年 vs 3 年），所以都会选择招供，最终各判 3 年。  
2. 但从集体理性看，双方沉默各判 1 年更好。  
3. 现实中的重复博弈、引入第三方惩罚或建立信任可以促进合作。  
4. 此困境展示了个人利益与集体利益的冲突。

Q：一道面试逻辑题：有两个沙漏，一个能计时 4 分钟，一个能计时 7 分钟，如何准确测量出 9 分钟？
A：1. 同时开始两个沙漏（4 分钟和 7 分钟）。

Q：一个简单的决策树：你参加一个游戏，面前有三个盒子，其中一个有奖品。你先选一个，主持人（知道哪个有奖）打开另一个空盒子，然后问你要不要换。你应该换吗？为什么？
A：1. 这是蒙提霍尔问题（Monty Hall problem）。  
2. 初始选中奖品的概率是 1/3，未选中的概率是 2/3。  
3. 主持人排除一个空盒子后，如果坚持不换，中奖概率仍是 1/3；如果换，中奖概率变为 2/3（因为最初选错时换就会对）。  
4. 所以应该换。  
5. 推理链：事件分解 + 条件概率，可以通过枚举所有可能验证。

Q：一个生活决策：你正在考虑是否要接受一个高风险高回报的创业机会。你如何用决策树分析？
A：1. 列出可能的结果：成功（获得高回报），失败（损失投入）。  
2. 估算成功概率 p 和失败概率 1-p，以及成功后的净收益，失败后的损失。  
3. 计算期望收益 = p * (成功收益) + (1-p) * (-损失)。  
4. 如果期望收益为正，且你能承受失败损失，可以考虑接受。  
5. 也可以考虑是否可以通过分阶段投入、寻找合伙人来降低风险。  
6. 进一步，可以构建决策树，考虑后续选项（如失败后是否还有出路）。  
7. 该过程体现了理性决策与风险偏好的结合。
